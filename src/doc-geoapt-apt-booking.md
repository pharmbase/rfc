# API Бронирования данных в geoapteka.ua/ru для аптечных сетей

## Соглашения
1. Протокол: [HTTP], [NATS].

2. Формат: [JSON].

3. Кодировка: [UTF-8] без [BOM].

3. Обмен данными происходит на основе идентификаторов компании Морион.

4. Код ответа сервера:
  * `200` - запрос выполнен успешно.
  * `403` - неверный логин или пароль в запросе.
  * `500` - внутренняя ошибка сервиса (в теле ответа будет содержаться описание ошибки).
  
5. Примеры осуществляются при помощи служебной программы [cURL].
  
## Принцип работы
АПИ предоставляет инструментарий, дающий возможность обрабатывать заказы как централизовано, так и распределяя заказы по торговым точкам, используя различный транспорт для получения заказов. Для этого предоставляется набор методов, которые имеют разную сложность, скорость и надежность доставки.


### Соединение
1. Тип соединения с АПИ - [HTTP] 

2. Тип соединения с [NATS] - [TLS]

3. Для соединения, вам понадобится *адрес*, а также *логин* и *пароль*, которые вы можете узнать у вашего менеджера.

### Обмен данными
1. На выбор предоставляются такие виды транспорта как [NATS], и самостоятельный забор заказов посредством [HTTP] запросов.

2. При использовании [NATS] соединения, АПИ ожидает ответа от клиентов в момент запроса, в случае, если ответ не получен в течении 5ти секунд, запрос считается неудачным.

3. При использовании метода на самостоятельный забор заказов, у сети есть 10 минут на то, что бы забрать заказ и 30 минут на его обработку. Если в отведенное время сеть не обработала заказ, он передается на внутреннюю службу обработки Morion, и в дальнейшем, будет недоступен для сети.

4. Предусмотрен вариант демонстрационного бронирования, при котором торговая точка получит специальный флаг, который информирует ее о том, что клиент не намерен реально бронировать товар.
В этом случае, торговая точка просто отдает данные, помечая ее соответствующим статусом, но реальное бронирование по остаткам не выполняет.

5. Для использования любых видов транспорта, кроме забора заказов по оффлайну, клиенту необходимо указать для необходимых идентификаторов торговых точек такие параметры как способ доставки, и адрес доставки. Если вид транспорта не указывается, по умолчанию используется метод самостоятельного забора заказов `/pop-order`. Методы для каждого вида транспорта отдельные и указаны ниже.


## Методы

### `/reg-nats`
Метод [HTTP]. Сообщает сервису название прослушиваемого канала [NATS]. Допустимые символы в названии канала: [`0-9`, `a-z`, `A-Z`, `-`, `_`].

Пример:
```sh
curl -X POST -T "order.json" -u login:password http://{addr}/reg-nats
```
Где: 
- `order.json` - данные в JSON-формате.
- `{addr}`     - адрес API.
- `login`      - логин (`booking` - для участия в бронировании).
- `password`   - пароль (ключ облачной авторизации).

Формат тела запроса:
```
[
 {
   "id_shop": "string", // Идентификатор торговой точки 
   "subject": "string"  // Название канала для данного идентификатора торговой точки     
 }
] 
```

Пример тела запроса:
```
[
 {
   "id_shop": "556546", 
   "subject": "my_secret_subject_1"      
 },
 {
   "id_shop": "122254", 
   "subject": "my_secret_subject_2"   
 }
] 
```

Указав единое значение `subject` для всех идентификаторов, возможно обрабатывать заказы "централизовано" (получать все на один канал).

### `/get-id-shop`
Метод [HTTP]. Позволяет получить идентификатор торговой точки для работы с API. Либо несколько идентификаторов сразу.

Пример:
```sh
curl -X POST -T "order.json" -u login:password http://{addr}/get-id-shop
```
Где: 
- `order.json` - набор данных JSON-формата.
- `{addr}`     - адрес API, который вы можете узнать у вашего менеджера.
- `login`      - логин HTTP аутентификации (`booking` - для участия в бронировании).
- `password`   - пароль HTTP аутентификации (ключ облачной авторизации).

Формат тела запроса:
```
[
 {
   "name": "string", // Название торговой точки
   "head": "string", // Название торговой сети (юр. лица)
   "addr": "string"  // Адрес торговой точки
 }
] 
```

Формат тела ответа при успешном запросе:
```
[
 {
   "name": "string", // Название торговой точки
   "head": "string", // Название торговой сети (юр. лица)
   "addr": "string", // Адрес торговой точки
   "id": "string"    // Идентификатор торговой точки
 }
] 
```

### `/test-order`
Метод [HTTP]. Позволяет самостоятельно составлять тестовый заказ для проверки функционала доставки сообщений.

Пример:
```sh
curl -X POST -T "order.json" -u login:password http://{addr}/test-order
```
Где: 
- `order.json` - набор данных JSON-формата.
- `{addr}`     - адрес API, который вы можете узнать у вашего менеджера.
- `login`      - логин HTTP аутентификации (`booking` - для участия в бронировании).
- `password`   - пароль HTTP аутентификации (ключ облачной авторизации).

Формат тела тестового запроса:
```
{
  "agent": "string",      // Маркер точки запроса, например "CorpName"
  "phone": "string",      // Телефон пользователя 
  "shops": [{
    "id_shop":  "string", // Идентификатор торговой точки
    "shipping": "string", // Способ доставки товара
    "data": [{
      "id": "string",     // Идентификатор товара
      "quant": 0.0,       // Количество
      "price": 0.0        // Цена
    }]
  }]
}
```
Где поле `"shipping"` описывает способ доставки заказа:
* `pickup`  - заказ клиент заберет самостоятельно.
* `deliver` - заказ необходимо доставить клиенту.

Обрабатываются тестовые заказы, теми же методами, что и реальные, отличаются от реальных наличием поля `"test": true` в теле запроса.

Пример тела тестового запроса:
```
{
  "agent": "CorpName",
  "phone": "380632670315",
  "shops": [{
        "id_shop":  "700555",
        "shipping": "pickup",
    "data": [{
        "id": "45600",
        "quant": 5.0,
        "price": 35.0
    },{
        "id": "500600",
        "quant": 1.0,
        "price": 153.45
    }]
},{     "id_shop":  "800900",
        "shipping": "pickup",
    "data": [{
        "id": "400800",
        "quant": 2.0,
        "price": 13.0
    }]
  }]
}
```

### `/pop-order`
Метод [HTTP]. Самостоятельный забор заказов для сети, которая участвует в бронировании. Получить можно как все заказы, так и выборочно, по конкретным торговым точкам.

Пример:
```sh
curl -X POST -T "order.json" -u login:password http://{addr}/pop-order
```
Где: 
- `order.json` - набор данных JSON-формата.
- `{addr}`     - адрес API, который вы можете узнать у вашего менеджера.
- `login`      - логин HTTP аутентификации (`booking` - для участия в бронировании).
- `password`   - пароль HTTP аутентификации (ключ облачной авторизации).

Формат тела запроса:
```
["string"] // Перечень идентификаторов торговых точек, по которым необходимо получить заказы
```

Пример тела запроса:
```
["111222", "333444", "5556667"]
```

Допускается вариант, при котором торговая сеть может централизовано получить все заказы для неё, в таком случае, в массиве указывается строка `"*"`.

Например:
```
["*"]
```

Формат тела ответа от АПИ при успешном запросе:
```
[
 {
   "phone":   "string",  // Телефон пользователя 
   "id_shop": "string",  // Идентификатор торговой точки 
   "test":    false,     // Флаг тестового запроса 
   
   "id_order": "string", // Номер брони
   "state":    "string", // Статус обработки запроса
   "shipping": "string", // Способ доставки товара

        "id": "string",  // Идентификатор товара

] 
```

* `pickup`  - заказ клиент заберет самостоятельно.

Поле `"state"` описывает текущее состояние брони:
* `New` - новый заказ, который ожидает обработки

Пример тела ответа при успешном запросе:
```
[
  {
    "phone":   "380365552233", 
    "id_shop": "800600", 
    "test":    false,     
    
    "id_order": "000001",
    "state":    "New",

        "id": "56548",
        "quant": 2.0,

        "id": "84655",
        "quant": 1.0,
        "price": 156.88

 },{
    "phone":   "380367773355", 
    "id_shop": "700100", 
    "test":    false,
```

### Прослушивание и бронирование
Метод [NATS]. Real-time соединение.
Данный метод используется для прослушивания запросов от клиентов в реальном времени, и обратного ответа по принципу Request–Response.
Для централизованного забора и бронирования заказов, сети достаточно одного соединения для прослушивания.
При распределенном способе получения заказов, устанавливается отдельное, уникальное соединение (например в каждой торговой точке).

Формат тела запроса отклиентов:
```
{
  "phone":    "string", // Телефон пользователя 
  "id_shop":  "string", // Идентификатор торговой точки 
  "test":     false,    // Флаг тестового запроса
  
  "id_order": "string", // Номер брони
  "shipping": "string", // Способ доставки товара
   "data": [{
      "id": "string",   // Идентификатор товара
      "quant": 0.0,     // Количество
      "price": 0.0      // Цена
   }]
}
```

Где поле `"shipping"` описывает способ доставки заказа:
* `pickup`  - заказ клиент заберет самостоятельно.
* `deliver` - заказ необходимо доставить клиенту.

Поле `state` описывает текущее состояние заказа:
* `Confirmed` - заказ принят, бронь зарезервирована.
* `Updated`   - запрос в аптеку не верен, повторите запрос с новыми данными (которые выдаются торговой точкой)

Пример тела запроса отклиентов:
```
{
  "phone":   "380365552233", 
  "id_shop": "800600", 
  "test":    false,
  
  "id_order": "866523",
  "shipping": "pickup",
    "data": [{
        "id": "56548",
        "quant": 2.0,
        "price": 15.40
    },{
        "id": "84655",
        "quant": 1.0,
        "price": 156.88
   }]
}
```

Формат тела ответа для отклиентов:
```
{
  "phone":   "string",   // Телефон пользователя 
  "id_shop": "string",   // Идентификатор торговой точки 
  "test":    false,      // Флаг тестового запроса
  
  "id_order": "string",  // Номер брони в аптеке
  "state":    "string",  // Статус обработки запроса
  "order_exp": 0,        // Срок истечения брони в аптеке, в формате Unixtime (в случае успешного бронирования)
  "shipping":  "string", // Способ доставки товара
    "data": [{
        "id": "string",  // Идентификатор товара
        "quant": 0.0,    // Количество
        "price": 0.0     // Цена
    }]
}
```

Пример тела ответа для отклиентов:
```
{
  "phone":   "380365552233", 
  "id_shop": "800600", 
  "test":    false, 
  
  "id_order":  "012345",
  "state":     "Confirmed",
  "order_exp": 11154655,
  "shipping":  "pickup",
    "data": [{
        "id": "56548",
        "quant": 1.0,
        "price": 35.10
    }]
}
```

В случае несовпадения цены, либо количества товара, отправляется *полный набор данных*, **с внесенными в них изменениями и соответствующим статусом заказа** в качестве ответа на запрос клиенту.
  
### `/upd-order`
Метод [HTTP]. Запрос на обновления данных по `заказу` в `конкретной` торговой точке.

Пример:
```sh
curl -X POST -T "order.json" -u login:password http://{addr}/upd-order
```
Где: 
- `order.json` - набор данных JSON-формата.
- `{addr}`     - адрес API, который вы можете узнать у вашего менеджера.
- `login`      - логин HTTP аутентификации (`booking` - для участия в бронировании).
- `password`   - пароль HTTP аутентификации (ключ облачной авторизации).

Формат тела запроса на обновления данных:
```
{
  "phone":   "string",   // Телефон пользователя 
  "id_shop": "string",   // Идентификатор торговой точки 
  "test":    false,      // Флаг тестового запроса  
  
  "id_order":  "string", // Номер брони в аптеке
  "state":     "string", // Статус обработки запроса
  "order_exp": 0,        // Срок истечения брони в аптеке, в формате Unixtime (в случае успешного бронирования)
  "shipping":  "string", // Способ доставки товара
    "data": [{
        "id": "string",  // Идентификатор товара
        "quant": 0.0,    // Количество
        "price": 0.0     // Цена
    }]
}
```

Где поле `"shipping"` описывает способ доставки заказа:
* `pickup`  - заказ клиент заберет самостоятельно.
* `deliver` - заказ необходимо доставить клиенту.

Поле `"state"` описывает текущее состояние брони:
* `Confirmed` - заказ подтвержден и зарезервирован в торговой точке.
* `Updated`   - заказ изменен со стороны торговой точки
* `Changed`   - заказ был изменен покупателем
* `Completed` - заказ был продан покупателю
* `Canceled`  - заказ был отменен торговой точкой, либо покупателем

Во всех вышеописанных случаях отправляется *полный набор данных*, **с внесенными в них изменениями и соответствующим статусом заказа**.

Пример тела запроса на обновления данных:
```
{
  "phone":   "380365552233", 
  "id_shop": "800600", 
  "test":    false, 
  
  "id_order":  "012345",
  "state":     "Confirmed",
  "order_exp": 11154655,
  "shipping":  "pickup",
    "data": [{
        "id": "56548",
        "quant": 1.0,
        "price": 35.10
    }]
}
```
